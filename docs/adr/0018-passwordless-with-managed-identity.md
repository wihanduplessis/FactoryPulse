# ADR-0018: Passwordless by default ŌĆö managed identity over vaulted credentials

- **Status:** Accepted
- **Date:** 2026-07-12
- **Deciders:** Project owner

## Context

The deployed system has four things that would conventionally be secrets:

1. The SQL connection string (username + password)
2. The container registry credentials (username + password)
3. The JWT signing key
4. The seeded admin password

The obvious design is "put all four in Key Vault." That is a fine answer, and it is the
answer most projects give. It stops one step short.

## Decision

**Eliminate the secrets that do not need to exist; vault only the two that genuinely
are secrets.**

| Value | Conventional | **What FactoryPulse does** |
|---|---|---|
| SQL connection | `User Id=sa;Password=ŌĆ”` | **Entra authentication.** `Authentication=Active Directory Default` ŌĆö **there is no password, because none exists.** `azureADOnlyAuthentication: true` on the server means a SQL login *cannot* be created. |
| Registry pull | Registry username + password | **Managed identity with `AcrPull`.** `adminUserEnabled: false` on the registry. |
| CI ŌåÆ Azure | Service principal secret | **OIDC federated credentials** (ADR-0020). |
| **JWT signing key** | ŌĆö | **Key Vault.** Irreducibly a secret. |
| **Seed admin password** | ŌĆö | **Key Vault.** Likewise. |

So the running application holds **exactly two secrets**, both read at startup from Key
Vault via its managed identity, and **nothing sensitive exists** in the Bicep templates,
in GitHub, or in any configuration file. The SQL connection string is a Bicep *output*
and sits in plain text ŌĆö safely, because it is worthless without an Entra identity that
already holds a role.

### How it fits together

A **user-assigned managed identity** (`id-factorypulse`) is created *before* the
container app that uses it. This is not incidental: a system-assigned identity cannot
exist before the app it belongs to, so it could not be granted roles that the app needs
in order to start. The user-assigned identity breaks that circular dependency.

It holds:
- `AcrPull` on the registry
- `Key Vault Secrets User` on the vault ŌĆö **read only.** The app cannot add, overwrite,
  or delete a secret. A compromised app could read the two secrets it already uses, and
  nothing more.
- A SQL database user created `FROM EXTERNAL PROVIDER`, in **`db_datareader` +
  `db_datawriter`** ŌĆö and deliberately **not** `db_owner` or `db_ddladmin`.

The human operator separately holds `Key Vault Secrets Officer` (to write the secrets)
and is the SQL server's Entra admin. Two roles, least privilege on both sides.

### The consequence that matters

**The application cannot alter its own schema.** Not "should not" ŌĆö *cannot*, enforced
by the database. This is what forces migrations out of application startup and into the
deployment pipeline (ADR-0019). The design makes the correct behaviour the only possible
behaviour, rather than trusting a future developer to remember.

`AZURE_CLIENT_ID` is set on the container so that `DefaultAzureCredential` selects the
right user-assigned identity; without it, the credential cannot choose, and both Key
Vault and SQL authentication fail.

## Alternatives considered

- **Everything in Key Vault.** Perfectly respectable. Rejected because "I removed two of
  the four secrets entirely, and vaulted the two that were real" is both a better answer
  and a better system: a secret that does not exist cannot leak, cannot be committed by
  accident, and never needs rotating.
- **Registry admin user.** One checkbox, and it puts a password back into GitHub,
  undoing the entire point of OIDC.
- **SQL admin password in Key Vault.** Works. Still leaves a password in the world.
- **System-assigned managed identity.** Simpler in the common case, but cannot be granted
  roles before the app exists. Rejected on the circular-dependency grounds above.

## Consequences

### Positive

- Two secrets in total, both generated by a cryptographic RNG, written straight to Key
  Vault, and never present in the repository, a config file, or a log.
- No credential rotation burden for the database or the registry, because there are no
  credentials.
- Least privilege is enforced by the platform, not by convention.
- The answer to "how do you handle secrets in production?" is a genuinely strong one.

### Negative / trade-offs

- **Local development still uses a SQL password** (the Docker SA account) and
  user-secrets. Entra auth is not available against a SQL Server container. So the app
  supports two authentication paths ŌĆö though the *code* is identical, because
  `DefaultAzureCredential` and the connection string absorb the difference.
- **Setup is more involved.** The managed identity must be created as a database user
  (`CREATE USER ŌĆ” FROM EXTERNAL PROVIDER`) inside the database ŌĆö an Azure role gets an
  identity *to* the database, it does not make it a user *in* the database. These are two
  separate permission systems, and forgetting the second produces a bewildering "login
  failed" against a perfectly configured identity.
- Debugging a failed managed-identity authentication is harder than debugging a wrong
  password.
