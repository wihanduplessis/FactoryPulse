# ADR-0002: Use GUID primary keys for entities

- **Status:** Accepted
- **Date:** 2026-07-04
- **Deciders:** Project owner

## Context

Every entity needs a primary key. FactoryPulse is a greenfield, cloud-oriented
application that may later involve distributed components, client-side ID
generation, and public-facing APIs. The two mainstream choices are sequential
integer identities (`int`/`bigint` IDENTITY) and GUIDs (`uniqueidentifier`).

## Decision

Entities will use **GUID (`Guid` / `uniqueidentifier`) primary keys**, exposed as
`public Guid Id { get; set; }`. IDs are generated by EF Core's built-in
**sequential GUID** value generator on insert.

## Alternatives considered

- **Integer identity keys** — smallest storage, best index locality, simplest to
  read. But values are sequential and guessable when exposed in URLs, require a
  database round-trip (or the row to be inserted) to obtain, and are awkward to
  merge across multiple data sources or generate on a client.
- **GUID keys** — chosen: generable anywhere (client or server) without
  coordination, non-sequential so they don't leak record counts or ordering,
  and painless if data is ever merged across systems. Common in enterprise .NET.

## Consequences

### Positive

- IDs can be created client-side or server-side without a DB round-trip.
- No sequential enumeration of records via the API surface.
- Simplifies any future distributed / multi-source scenario.

### Negative / trade-offs

- A `uniqueidentifier` is 16 bytes vs 4 for an `int`, enlarging the key and any
  indexes that include it.
- Random GUIDs cause clustered-index fragmentation. **Mitigated** by using EF
  Core's sequential GUID generator, which produces near-ordered values, so at
  this project's scale fragmentation is a non-issue.

### Follow-ups

- If a table ever reaches a scale where key size/fragmentation measurably hurts,
  revisit for that specific table (e.g. `NEWSEQUENTIALID()` or a keyset design).
